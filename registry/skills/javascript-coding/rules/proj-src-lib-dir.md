# proj-src-lib-dir

> Use `src/` for source code and `dist/` for build output

## Why It Matters

Mixing source code, build output, configuration, and tests at the project root creates a cluttered directory. The `src/` / `dist/` separation makes it clear what is source (edited by developers) vs output (generated by tools), simplifies tooling configuration, and prevents accidentally editing generated files.

## Bad

```
my-project/
  server.js
  server.test.js
  database.js
  config.js
  utils.js
  logger.js
  package.json
  .env
  node_modules/
```

## Good

```
my-project/
  src/
    server.js
    database.js
    config.js
    utils/
      logger.js
    routes/
      users.js
      orders.js
    services/
      user-service.js
    index.js
  tests/
    server.test.js
    user-service.test.js
  dist/              # Build output (gitignored)
  package.json
  .env.example
  .gitignore
```

## Standard Layout

```
src/
  index.js           # Entry point — exports public API or starts server
  config.js          # Configuration
  routes/            # HTTP route handlers
  services/          # Business logic
  repositories/      # Data access layer
  middleware/         # Express/HTTP middleware
  utils/             # Shared utilities
  types.js           # JSDoc type definitions

tests/
  unit/
  integration/

dist/                # Build output — gitignored
```

## When Exceptions Apply

For very small projects (< 5 files), a flat structure is fine. Introduce `src/` when the project grows beyond a handful of files.
