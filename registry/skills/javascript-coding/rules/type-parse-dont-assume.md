# type-parse-dont-assume

> Parse JSON and API responses with schema validation — never assume the shape

## Why It Matters

External data from APIs, message queues, and file I/O has no type guarantees. Assuming a response shape leads to `TypeError: Cannot read properties of undefined` when the API changes or returns unexpected data. Schema validation at the boundary ensures data matches expectations before it enters your application.

## Bad

```js
// Assuming the response shape — fragile
const response = await fetch('/api/users');
const users = await response.json();
users.forEach(user => {
  console.log(user.address.city);  // TypeError if address is null
});
```

## Good

```js
import { z } from 'zod';

const UserSchema = z.object({
  id: z.string(),
  name: z.string(),
  email: z.string().email(),
  address: z.object({
    city: z.string(),
    country: z.string(),
  }).nullable(),
});

const UserListSchema = z.array(UserSchema);

async function fetchUsers() {
  const response = await fetch('/api/users');
  const data = await response.json();
  return UserListSchema.parse(data);  // Validates and throws on mismatch
}

const users = await fetchUsers();
for (const user of users) {
  console.log(user.address?.city ?? 'Unknown');  // Safe access after validation
}
```

## Validate, Then Trust

```js
// Parse at the boundary, trust internally
async function loadConfig(path) {
  const raw = JSON.parse(await readFile(path, 'utf8'));
  return ConfigSchema.parse(raw);  // After this, config is known-valid
}

// Internal code trusts the validated data
function startServer(config) {
  // No further validation needed — config was parsed at the boundary
  app.listen(config.port);
}
```

## When Exceptions Apply

For data from fully trusted sources (generated by your own build pipeline, files committed to the repo), validation can be lighter. But API responses and user input should always be validated.
