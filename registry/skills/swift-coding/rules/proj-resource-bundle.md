# proj-resource-bundle

> Use SPM resource bundles for assets instead of ad hoc paths

## Why It Matters

Loading assets with a hardcoded `Bundle.main` path assumes the resource lives in the app's main bundle, which breaks the moment the code is reused from a Swift package, a framework, an extension, or a test target — each has its own bundle, and `Bundle.main` inside a package resolves to whatever executable links it, not the package itself. SPM's `resources:` declaration on a target generates a `Bundle.module` reference scoped correctly to that target, so the same code loading the same asset works identically whether it's linked into the app, a widget extension, or a unit test.

## Bad

```swift
// Inside a Swift package target — assumes it's always the main app bundle
struct IconProvider {
    static func icon(named name: String) -> UIImage? {
        UIImage(named: name, in: Bundle.main, compatibleWith: nil)
        // breaks when this package is linked into a widget extension or unit test target
    }
}
```

## Good

```swift
// Package.swift
.target(
    name: "IconKit",
    resources: [.process("Resources")]
)
```

```swift
// IconProvider.swift — inside the IconKit target
struct IconProvider {
    static func icon(named name: String) -> UIImage? {
        UIImage(named: name, in: .module, compatibleWith: nil)
        // Bundle.module is generated by SPM and always resolves to this target's bundle
    }
}
```

## `.process` vs `.copy`

Use `.process(_:)` for resources that should go through the platform's standard asset pipeline (asset catalogs, localized strings, storyboards) — SPM applies compiler/asset optimizations. Use `.copy(_:)` for files that must be included verbatim, byte-for-byte, with no transformation (raw JSON fixtures, certificates, non-standard file formats):

```swift
.target(
    name: "IconKit",
    resources: [
        .process("Resources/Assets.xcassets"),   // goes through asset catalog compilation
        .copy("Resources/config.json")           // included as-is, no processing
    ]
)
```

Localized resources also need `.process`, with locale subfolders (`Resources/en.lproj/Localizable.strings`) resolved automatically by `Bundle.module` based on the user's locale — no manual bundle path juggling required.
