# embed-redis-scripting

> Follow Redis Lua scripting constraints: determinism, `KEYS`/`ARGV`, no `os`/`io`

## Why It Matters

Redis executes Lua scripts (`EVAL`/`EVALSHA`, or Redis Functions since Redis 7) inside a restricted, sandboxed interpreter with no `os`, no `io`, no `require`, and a hard requirement that scripts be deterministic (since replication and the append-only file replay the *script*, not just its effects, on older Redis versions — and even on newer versions with effects-replication, non-determinism still causes replica drift in some configurations). Scripts must also declare their touched keys via `KEYS`, not hardcode them, so Redis Cluster can route them correctly.

## Bad

```lua
-- Hardcodes a key name instead of using KEYS[], breaking Cluster routing
-- and Redis's ability to verify which keys a script touches
local value = redis.call("GET", "user:1000")

-- Uses a non-deterministic source of randomness/time directly -- can cause
-- primary/replica or AOF-replay divergence
local token = tostring(math.random())
redis.call("SET", "session:" .. token, "active")

-- os/io are not available in the Redis Lua sandbox at all -- this errors
-- local f = io.open("/tmp/log", "a")
```

## Good

```lua
-- Declare touched keys via KEYS, values via ARGV -- required for Cluster
-- routing and for Redis to reason about the script's key access
-- EVAL "return redis.call('GET', KEYS[1])" 1 user:1000
local value = redis.call("GET", KEYS[1])

-- For anything needing "randomness", derive it deterministically from
-- data already passed in, or use Redis's own deterministic replication-safe
-- helpers where available, rather than math.random()/os.time()
local session_id = ARGV[1]  -- generated by the CALLING application, passed in
redis.call("SET", "session:" .. session_id, "active")

-- Use redis.error_reply / redis.status_reply for structured script results
if redis.call("EXISTS", KEYS[1]) == 0 then
  return redis.error_reply("key does not exist")
end
return redis.call("GET", KEYS[1])
```

## Version Note

Redis's embedded Lua has historically been based on Lua 5.1; check your specific Redis version's documentation for exact sandbox restrictions and available `redis.*` helper functions, and never assume standard-library functions beyond the documented allowlist are present.
