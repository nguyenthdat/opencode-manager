# type-bridge-method-awareness

> Understand bridge methods and erasure implications

## Why It Matters

The compiler silently generates synthetic "bridge methods" whenever a generic method is overridden with a more specific parameter type, so that erasure-based overriding still works correctly with the JVM's invocation rules. Not understanding bridge methods leads to confusing bugs with reflection (`getDeclaredMethods()` returning extra entries), `instanceof` surprises on `Method` objects, and mysterious `ClassCastException` from libraries that use raw types internally.

## Bad

```java
interface Adder<T> {
    void add(T item);
}

class IntAdder implements Adder<Integer> {
    @Override
    public void add(Integer item) {
        System.out.println("added " + item);
    }
}

// Reflection code that assumes there's exactly one add() method
IntAdder adder = new IntAdder();
for (Method m : adder.getClass().getDeclaredMethods()) {
    if (m.getName().equals("add")) {
        // BUG: this actually finds TWO methods:
        //   add(Integer)          <- the real one
        //   add(Object) bridge    <- synthetic, generated by the compiler
        System.out.println(m.getParameterTypes()[0]); // sometimes prints Object, not Integer
    }
}

// Raw-typed access can invoke the bridge and cause a surprising CCE
Adder raw = adder;
raw.add("not an integer"); // compiles due to raw type, throws ClassCastException
                            // inside the synthetic bridge method, not in add() itself
```

## Good

```java
interface Adder<T> {
    void add(T item);
}

class IntAdder implements Adder<Integer> {
    @Override
    public void add(Integer item) {
        System.out.println("added " + item);
    }
}

IntAdder adder = new IntAdder();
for (Method m : adder.getClass().getDeclaredMethods()) {
    // Filter out compiler-generated bridge methods explicitly
    if (m.getName().equals("add") && !m.isBridge()) {
        System.out.println(m.getParameterTypes()[0]); // reliably prints Integer
    }
}

// Never expose or assign to raw types - keeps the compiler enforcing add(Integer)
Adder<Integer> typed = adder;
// typed.add("not an integer"); // now a compile error, caught long before runtime
```

## What's Actually Happening

For `IntAdder`, the compiler emits two methods in the `.class` file:

- `add(Integer)` - the method you wrote
- `add(Object)` - a synthetic bridge that casts its argument to `Integer` and delegates to `add(Integer)`

This exists because at the bytecode level, `Adder.add(Object)` (post-erasure) must have a concrete override in `IntAdder`, even though your source only defines `add(Integer)`. Always call `Method.isBridge()` when writing reflection code that enumerates methods on generic types, and never rely on raw-typed references in application code, since they route calls through these bridges without warning.
