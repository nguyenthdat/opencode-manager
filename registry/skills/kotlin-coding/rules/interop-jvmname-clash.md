# interop-jvmname-clash

> Use `@JvmName` to resolve JVM signature clashes from Kotlin-specific features

## Why It Matters

Features with no direct JVM equivalent — extension functions whose erased signatures collide, or property/method name collisions — can produce a "platform declaration clash" that fails compilation. `@JvmName` renames the generated bytecode symbol to disambiguate without changing the Kotlin-visible name.

## Bad

```kotlin
// platform declaration clash: erasure makes both functions look like
// fun filterValid(List): List
fun List<String>.filterValid(): List<String> = filter { it.isNotBlank() }
fun List<Int>.filterValid(): List<Int> = filter { it > 0 } // ERROR: clashes with the above after erasure
```

## Good

```kotlin
@JvmName("filterValidStrings")
fun List<String>.filterValid(): List<String> = filter { it.isNotBlank() }

@JvmName("filterValidInts")
fun List<Int>.filterValid(): List<Int> = filter { it > 0 } // distinct JVM name, same Kotlin call-site name
```

## Property vs Function Name Collisions

```kotlin
class Person(private val _name: String) {
    val name: String
        @JvmName("getPersonName") // avoids clashing with a same-named function elsewhere in the class
        get() = _name
}
```

## Renaming for Java Ergonomics

```kotlin
class Rectangle(val width: Int, val height: Int) {
    @JvmName("area") // "getArea" would be the compiler's default; "area()" reads better from Java
    fun computeArea(): Int = width * height
}
```
