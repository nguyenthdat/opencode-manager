# interop-throws-checked-exceptions

> Declare `@Throws` so Java callers see checked exceptions

## Why It Matters

Kotlin has no checked exceptions, so a function that throws `IOException` compiles without any indication in its signature. Java callers won't be forced by the compiler to catch or declare it, silently losing a part of the API contract that Java developers rely on.

## Bad

```kotlin
class FileReaderService {
    fun readConfig(path: String): String {
        return File(path).readText() // can throw IOException, but Java sees no `throws` clause
    }
}
```

```java
// Java compiles fine with no forced try/catch - IOException surprises at runtime
String config = new FileReaderService().readConfig("config.txt");
```

## Good

```kotlin
class FileReaderService {
    @Throws(IOException::class)
    fun readConfig(path: String): String {
        return File(path).readText()
    }
}
```

```java
// Java compiler now requires a try/catch or a `throws IOException` declaration
try {
    String config = new FileReaderService().readConfig("config.txt");
} catch (IOException e) {
    // handle
}
```

## Kotlin Callers Are Unaffected

`@Throws` only affects the generated bytecode's `throws` clause for Java interop; Kotlin code calling the same function is not required to catch anything, preserving Kotlin's unchecked-exception model.
